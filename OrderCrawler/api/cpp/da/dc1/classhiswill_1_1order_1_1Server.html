주문기록 크롤러 서버.

Server 는 OrderCrawler를 실시간 (클라우드) 서버로 만들어주는 클래스이다. Flex 클라이언트의 접속을 수용해 이에 대응하는 Client 객체를 생성하고, replyData 메소드를 전체 서버 단위의 작업을 수행하고, sendData 메소드를 통하여 서버에 접속해있는 전체 Flex 클라이언트들에게 메시지를 보낸다.

Server 는 본래 네트워크 정의 내 서버의 역할에 더불어, contractorArray 를 통하여 외부 계약사 및 각 계약사별 거래정보 보유하며, Flex 클라이언트와의 연동 뿐 아니라, 데이터베이스 입출력 기능을 위한 SQL 인터페이스 sqli 또한 가지고 있다 (단, 객체를 소유한다는 의미이지, 관리나 프로세스를 총괄한다는 의미는 아님).

  • IServer: http://samchon.github.io/framework/api/cpp/d4/de8/classsamchon_1_1protocol_1_1IServer.html
  • IProtocol: http://samchon.github.io/framework/api/cpp/d4/dcf/classsamchon_1_1protocol_1_1IProtocol.html
할일:
RWMutex가 시간을 너무 많이 잡아먹는 문제가 있다. contractorArray 전체 단위에 대한 락을 수행하기 때문임

OrderCrawler 서버는 본래, 자체적으로 데이터를 동기화하기 때문에, DB에서 데이터를 부르는 일은 서버가 구동되기 시작하는 시점, 초기화 시점에만 딱 한 번 필요하다. 하지만, 허용 컴포넌트의 변경은 본 서버가 아닌 외부에서 사용자가 DB에 직접 액세스하여 이루어진다.

이 때문에 허용 컴포넌트의 갱신을 위해서라도 부득이 DB에서 데이터를 새로 받아와야 하며, 이 과정이 크롤링 때 함께 이루어지도록 해 놓았다. 하지만, 추후 관리자 페이지에 허용 컴포넌트를 관리하는 기능이 생길 경우, DB에서 기록을 주기적으로 불러오는 이 기능은 사라질 것이다.

플래시의 cross domain policy를 담당하는 서버.

플래시(swf)는 외부 도메인의 서버에 접속할 때, 크로스 도메인 정책을 정의한 CrossDomainPolicy.xml 파일을 요구한다. 이 파일은 도메인의 루트 경로에 위치해야 하는데, 현재의 OrderCralwer는 클라우드 서버로써, 따로이 도메인을 가지는 것이 아니다.

이러한 경우엔 플래시는 다시 843 포트로 크로스 도메인 정책을 정의한 XML을 요청하는데, FlashPolicyServer 객체가 바로 이 크로스 도메인 정책을 전송해주는 서버 역할을 한다.
